## 定義のアップデート

大量・多様なデータを破綻なくユーザーへ届ける。

# 1. コンテンツ駆動の深化
## ファイル管理からデータ基盤へ

---

## Content Layer API & Live Collections

ファイルベース管理の限界（メモリ不足・ビルド停止）を解消。

**Content Layer API** は、外部データを「内蔵SQLiteキャッシュ」に取り込み、型定義を強制。大規模サイトでも省メモリ・高速処理を維持。

**Live Collections** により、外部CMSの更新をプレビューへ反映。HMRにより、リアルタイムで変更を確認。

# 2. パフォーマンスの再定義
## Islands から Server Islands へ

---

## Server Islands

「表示速度」と「ユーザー最適化」のトレードオフを解消。SSR（表示遅延）とSSG（更新不可）の課題を解決。

**Server Islands** は、ページ全体を「静的な外枠」として返却。ログイン情報などの「動的パーツ」をサーバー側で遅延レンダリングして挿入。速度を維持したままパーソナライズを両立。

# 3. アプリケーション基盤の強化

## SaaS開発を支える標準化

---

## Astro Actions & Standardized Sessions

APIルート設計や手動の型定義によるバグ・工数を削減。

**Astro Actions** は、サーバー処理を「型安全な関数」として呼び出し。入力チェックから結果取得までを保証。

**Standardized Sessions** はログイン状態管理を共通化。デプロイ先を問わず、権限チェックや表示のチラつきを解消。開発者が機能開発に集中するためのインフラ。

# 4. 開発基盤の刷新
## Vite Environment API への移行

---

## Vite Environment API

### What：実行環境の分離と独立
- **マルチランタイム対応**: ブラウザ、サーバー、エッジなどの異なる実行環境を、一つの開発サーバー内で独立して管理。
- **複数のモジュールグラフ**: 環境ごとに依存関係を管理。SSR用コードとクライアント用コードの混在による競合を解消。

### Why：複雑なレンダリングパターンの安定化
- **環境制約の解決**: Node.js 固有の API とエッジランタイム用のコードが共存する際の、開発サーバーの挙動を安定化。
- **Server Islands の基盤**: 静的・動的、および異なる環境で動くコンポーネントが混在する Astro のアーキテクチャを、本番環境と一致した挙動で開発可能に。
